# Plan: Waitlist mit Resend live schalten (nach Pull aktualisiert)

## Ausgangslage (Repo-Check nach Pull)
- In `src/pages/index.astro` gibt es jetzt bereits eine Waitlist-Sektion mit Formular (`#waitlist-form`).
- Der aktuelle Submit ist rein clientseitig und simuliert Erfolg via `localStorage` (`lemonspace-waitlist`), ohne Backend-Call.
- `astro.config.mjs` hat inzwischen `@astrojs/react`, aber weiterhin **kein** `output: "server"` und keinen Deployment-Adapter.
- Es gibt noch keine API-Routen unter `src/pages/api/`.
- Waitlist-i18n ist schon vorhanden in `src/i18n/en.json` und `src/i18n/de.json`.

## Relevante Context7-Erkenntnisse (latest docs)
- **Resend SDK**: `new Resend(process.env.RESEND_API_KEY)` und `{ data, error }` Pattern.
- **Resend Contacts**: `resend.contacts.create({ email, firstName, lastName, properties })` für strukturierte Lead-Erfassung.
- **Astro API-Routen**: POST-Handling über `request.formData()` in `src/pages/api/*.ts`.
- **Astro Voraussetzung**: Für sichere serverseitige Secret-Nutzung muss Astro als Server-Output mit Adapter laufen.

## Umsetzungsplan

1. **Server-Runtime aktivieren**
   - In `astro.config.mjs` auf `output: "server"` wechseln.
   - Passenden Adapter für das tatsächliche Deployment hinzufügen (Vercel/Netlify/Node).
   - Ziel: `RESEND_API_KEY` bleibt serverseitig und landet nicht im Client-Bundle.

2. **Resend anbinden**
   - `resend` als Dependency ergänzen.
   - Kleine Utility in `src/lib/` für zentralen Resend-Client erstellen.
   - Env-Variablen definieren:
     - `RESEND_API_KEY` (required)
     - `RESEND_WAITLIST_AUDIENCE_ID` (optional, falls für eure Contacts-Struktur benötigt)
     - optional `WAITLIST_NOTIFY_TO` für interne Benachrichtigung.

3. **API-Endpunkt für Waitlist erstellen**
   - Neue Datei `src/pages/api/waitlist.ts` mit `POST`-Handler.
   - Input-Flow:
     - `request.formData()` lesen
     - `email` validieren (required + Syntax)
     - Honeypot-Feld prüfen (bot mitigation)
   - Resend-Flow:
     - Primär `contacts.create(...)` für echte Registrierung in Resend.
     - Dedupe tolerant behandeln (bereits vorhandene Adresse => `200` mit neutraler Success-Message).
   - API-Antwort standardisieren: `{ ok: boolean, message: string }`.

4. **Bestehendes Formular von Mock auf echtes Backend umstellen**
   - `src/pages/index.astro`:
     - `form` mit `method="POST"` und `action="/api/waitlist"` ausstatten.
     - Email-Input mit `name="email"` ergänzen (aktuell fehlt `name`).
     - Honeypot-Input hinzufügen.
   - Waitlist-Script anpassen:
     - statt `localStorage`-Simulation auf `fetch('/api/waitlist')` umstellen,
     - Erfolgs-/Fehlermeldungen aus API anzeigen,
     - bestehende i18n-Texte weiterverwenden.
   - Progressive Enhancement beibehalten (POST funktioniert auch ohne JS).

5. **i18n nur gezielt erweitern**
   - Bereits vorhandene Keys in `src/i18n/en.json` und `src/i18n/de.json` nutzen.
   - Nur fehlende States ergänzen (z. B. generic server error, submitting).

6. **Doku aktualisieren**
   - `README.md` um Setup für Waitlist ergänzen:
     - benötigte Env-Variablen,
     - lokale Test-Calls,
     - Hinweis auf Resend Domain-/Sender-Verifizierung (für Notification-Mail-Fallback).

7. **Verifikation**
   - `pnpm build` erfolgreich.
   - Endpoint-Fälle prüfen:
     - valid email => success,
     - invalid email => 400,
     - duplicate email => 200 (idempotent),
     - honeypot gesetzt => block/ignore.
   - UI-Check (EN/DE, mobile/desktop, theme toggle beeinflusst Waitlist nicht).

## Technische Entscheidungen (Default)
- **Default**: Waitlist-Registrierung via Resend Contacts (statt nur pseudo-localStorage oder reine Notification-Mail).
- **Fallback**: Optional interne Notification-Mail pro Signup (nur wenn `WAITLIST_NOTIFY_TO` gesetzt).
- **Datensparsamkeit**: Nur E-Mail als Pflichtfeld.

## Risiken / Stolpersteine
- Falscher Adapter für das echte Hosting führt zu Deployment-Problemen.
- Dedupe-Verhalten von Contacts muss robust gegen bereits vorhandene E-Mails sein.
- Bei zusätzlicher Mail-Benachrichtigung muss Sender-Domain in Resend verifiziert sein.

## Nächster Schritt: kurze Bestätigungsmail an Waitlist-Signups

### Ziel
- Nach erfolgreichem `contacts.create(...)` eine kurze Transaktionsmail an die eingetragene Adresse senden.

### Context7-Notizen (Resend)
- `resend.emails.send(...)` benötigt `from`, `to`, `subject` und `html` oder `text`.
- `from` muss auf einer in Resend verifizierten Domain liegen (sonst Domain-Mismatch/403).

### Geplante Umsetzung
1. **Env ergänzen**
   - `RESEND_FROM_EMAIL` (required, z. B. `LemonSpace <hello@updates.lemonspace.io>`)
   - optional `RESEND_REPLY_TO`.

2. **Mailversand in Waitlist-Endpoint ergänzen**
   - In `src/pages/api/waitlist.ts` nach erfolgreicher Contact-Erstellung:
     - `await resend.emails.send({...})` mit kurzer Confirmation-Mail.
   - Inhalt bewusst minimal:
     - Betreff: "You're on the LemonSpace waitlist"
     - 2-3 kurze Zeilen + kein Marketing-Overload.

3. **Fehlerstrategie festlegen**
   - Empfehlungs-Default: Contact-Erfolg bleibt `200`, auch wenn Bestätigungsmail fehlschlägt (nur Logging).
   - Begründung: Waitlist-Eintrag ist der primäre Business-Zielpfad.

4. **i18n-Entscheidung**
   - Start mit englischer Mail (schnellster Rollout) oder DE/EN abhängig von `lang` aus Formular.

5. **Verifikation**
   - Signup erzeugt Contact in Audience.
   - Signup triggert exakt eine Mail an dieselbe Adresse.
   - Duplicate-Signup verschickt standardmäßig keine zweite Mail (idempotentes Verhalten).

### Offene Entscheidung
- Entscheidung getroffen: Bestätigungsmail **nur beim ersten Eintrag** versenden; bei Duplicate keine erneute Mail.
