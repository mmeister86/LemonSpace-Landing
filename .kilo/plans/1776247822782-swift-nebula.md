# Plan: Pricing modularisieren, ausblenden, Waitlist unter Hero prominent machen

## Zielbild
- Die Pricing-Sektion wird als eigenes Modul ausgelagert.
- In dieser Landing-Variante wird Pricing nicht angezeigt.
- Direkt unter dem Hero erscheint eine zusaetzliche, deutlich prominentere Waitlist-Eintragung.
- Die bestehende Waitlist-Sektion weiter unten bleibt erhalten ("zusaetzlich"), aber ohne technische Duplikatsprobleme.

## Bestand (kurz)
- Pricing ist aktuell inline in `src/pages/index.astro` (`<section id="pricing">` plus Nav-Links auf `#pricing`).
- Waitlist-Form gibt es aktuell einmal in `src/pages/index.astro` mit festen IDs (`waitlist-form`, `waitlist-submit`, `waitlist-message`).
- Das Waitlist-Script bindet derzeit exakt diese eine Instanz.

## Umsetzungsplan

1. **Pricing als Modul extrahieren**
   - Neue Komponente anlegen, z. B. `src/components/PricingSection.astro`.
   - Bestehendes Pricing-Markup 1:1 aus `src/pages/index.astro` auslagern, inklusive `Check`-Icon-Nutzung.
   - API/Props minimal halten (z. B. `id = "pricing"` optional), damit die Sektion spaeter leicht wieder aktiviert werden kann.

2. **Pricing in dieser Version deaktivieren**
   - In `src/pages/index.astro` ein Seiten-Flag setzen, z. B. `const showPricing = false`.
   - Pricing-Komponente nur konditional rendern.
   - Desktop- und Mobile-Navigation ebenfalls an `showPricing` koppeln, damit keine toten `#pricing`-Links bleiben.

3. **Waitlist-Form als wiederverwendbares Muster vorbereiten**
   - Die Form-Struktur so umbauen, dass mehrere Instanzen moeglich sind (statt harter Einzel-IDs).
   - Entweder als kleine Komponente (empfohlen, z. B. `src/components/WaitlistSignup.astro`) oder als sauberes wiederverwendbares Markup in `index.astro`.
   - Jede Instanz bekommt lokale Hooks via `data-*` Attribute (z. B. `data-waitlist-form`, `data-waitlist-submit`, `data-waitlist-message`) statt globaler IDs.

4. **Prominente Waitlist direkt unter Hero einbauen**
   - Neue Section unmittelbar nach `<Hero152 />` einfuegen, mit `id="waitlist"` als primarem Anchor-Ziel fuer CTA-Links.
   - Visuell deutlich hervorheben (groesserer Headline-Block, kontrastreicher Hintergrund/Border/Shadow, staerkeres Formular-Layout, aber im bestehenden Warm-Organic-Stil).
   - Bestehende untere Waitlist-Sektion behalten, ggf. als sekundaere Variante (kompakter), damit "zusaetzlich" klar umgesetzt ist.

5. **Waitlist-JS auf mehrere Formulare erweitern**
   - In `src/pages/index.astro` das Script von Einzel-Element-Selektoren auf `querySelectorAll` umstellen.
   - Submit-Handling pro Formular instanzieren (eigener Button-Loading-State, eigene Message-Ausgabe).
   - `lang` Hidden-Input in jeder Instanz setzen.
   - Query-Parameter-Status (`?waitlist=...`) auf die primaere Instanz unter dem Hero routen und danach aus der URL entfernen (bestehendes Verhalten beibehalten).

6. **Styling fuer Prominenz und Responsiveness nachziehen**
   - In `src/styles/global.css` gezielte Klassen fuer die neue Hero-nahe Waitlist ergaenzen (keine generischen Dashboard-Patterns).
   - Mobile zuerst pruefen: klare vertikale Hierarchie, grosse Touch-Targets, kein Layout-Shift.
   - Dark-Theme mitdenken, aber Light bleibt visuelle Leitrichtung.

7. **i18n und Copy konsistent halten**
   - Vorhandene Keys aus `src/i18n/en.json` und `src/i18n/de.json` wiederverwenden, sofern moeglich.
   - Nur bei Bedarf neue Keys fuer die prominente Variante ergaenzen (z. B. eigener Upper-Funnel-Headline-Text).

## Verifikation
- `pnpm build` muss sauber durchlaufen.
- Navigation: kein sichtbarer Link mehr auf `#pricing`, CTA-Links springen auf die neue Waitlist unter dem Hero.
- Beide Waitlist-Formulare funktionieren:
  - invalid -> validierte Fehlermeldung,
  - success -> Erfolgsmeldung + Reset,
  - already -> neutrale "already on list" Meldung.
- EN/DE Umschaltung aktualisiert beide Formularinstanzen korrekt.
- Mobile/desktop und light/dark visuell konsistent.

## Deliverables
- `src/components/PricingSection.astro` (neu)
- `src/pages/index.astro` (Pricing-Flag, neue prominente Waitlist, Script-Refactor)
- optional `src/components/WaitlistSignup.astro` (neu, empfohlen)
- `src/styles/global.css` (Prominence-/Layout-Klassen)
- optional kleine i18n-Ergaenzungen in `src/i18n/en.json` und `src/i18n/de.json`
